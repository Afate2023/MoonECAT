# MoonECAT AI 项目路线图

## 1. 项目目标摘要

本路线图基于 [MoonECAT项目申报书.md](e:\Repos\EtherCAT\MoonECAT\MoonECAT项目申报书.md) 整理，面向人和 AI 协同执行，目标是把申报书中的能力边界转成可持续推进的实施计划。

- 项目定位：基于 MoonBit 的 EtherCAT 主站库，采用 `Library API + CLI(scan/validate/run)` 双入口。
- 核心目标：在内存安全前提下实现轻量实时、统一抽象、多平台部署和工程化工具链闭环。
- 目标平台：Linux、Windows、RTOS/Embedded、Extism 宿主方向，按阶段逐步落地，不承诺首版一次完成。
- 当前承诺范围：以 ETG.1500 `Class B` 核心能力为主。
- In Scope：OSAL/HAL、多后端网卡适配、Ethernet II 帧封装、ESM 状态机、位置寻址与别名寻址、CoE/SDO、PDO 周期交换、SII/ESI 解析、FMMU/SM 自动配置、DC PLL、周期调度、事务推进。
- Out of Scope：`EoE`、`FoE`。
- 主要使用场景：
  - 本地调试和测试从站的 `CLI`。
  - 面向嵌入式和上层系统集成的稳定库接口。
  - 后续面向在线诊断工具的 Extism 插件形态。

## 2. 总体实施策略

- 推进原则：坚持“核心纯逻辑 + 平台薄适配 + CLI/测试外壳”，避免协议逻辑和平台细节耦合。
- 架构落点：
  - `Platform HAL` 负责网卡收发、计时、任务调度、文件和抓包输出。
  - `Protocol Core` 负责帧编解码、PDU 管线、寻址、拓扑、ESM 和错误模型。
  - `Mailbox & Config` 负责 CoE/SDO、SII/ESI 解析、FMMU/SM 计算。
  - `Runtime Orchestrator` 负责 PDO 周期调度、事务推进、超时与背压。
  - `Product Surface` 负责库接口和 `scan/validate/run` CLI。
- 实施原则：
  - 先打通最小通信闭环，再扩展配置、状态机和诊断能力。
  - 先定义跨平台最小接口，再逐个补后端，避免平台实现倒逼核心接口反复重写。
  - 先保证热路径预分配和少分配，再补充高级功能，降低实时性回归风险。
  - CLI 与库必须共享同一套核心逻辑，不接受双实现分叉。

## 3. 分阶段路线图

### 阶段 1：基础骨架与包结构

目标：
- 建立可扩展的包结构、命名和模块边界。
- 明确 HAL、核心协议、配置事务、运行时、CLI 的分层接口。
- 建立最小测试骨架和文档入口。

核心任务：
- 设计 MoonBit 包布局和基础类型定义。
- 抽象 HAL 最小接口，覆盖收发、计时、调度、文件访问、错误码。
- 定义核心错误模型、日志/诊断事件模型、配置对象基础结构。
- 建立 `Library API` 和 `CLI` 的入口占位与公共依赖关系。
- 建立用于回归的最小测试目录和样例夹具组织方式。

验收标准：
- 仓库具备稳定包结构，后续功能无需大规模重排目录。
- HAL、Protocol Core、Mailbox、Runtime、CLI 的边界可通过接口和类型体现。
- 至少存在一组可编译的最小示例或空实现测试，确保骨架成立。
- 所有后续阶段都能明确挂靠到已有包结构和接口层次。

延后项：
- 不实现真实网卡后端。
- 不实现完整 EtherCAT 帧收发。
- 不承诺 CLI 命令可用，只保留入口规划。

### 阶段 2：HAL 与帧/PDU 最小闭环

目标：
- 建立 EtherCAT 帧编解码和 PDU 管线。
- 打通 Mock 或单平台后端下的最小收发闭环。
- 为后续状态机和扫描能力提供通信基础。

核心任务：
- 实现 Ethernet II 与 EtherCAT 基本帧结构的构造、解析和校验。
- 实现 PDU 编码、解码、地址模型和缓冲区管理。
- 提供首个可运行后端，优先 `Mock Loopback`，其次 Linux 原型。
- 建立发送、接收、超时、错误返回的统一处理流程。
- 加入热路径分配控制规则，验证核心数据结构可预分配。

验收标准：
- 在 Mock 或单平台环境中可完成一轮可重复的 EtherCAT 帧/PDU 收发测试。
- 帧和 PDU 相关单元测试覆盖正常路径和基本错误路径。
- HAL 接口足以支撑真实网卡后端接入，无需回退修改核心编码接口。
- 最小闭环不依赖 CLI，可由测试直接驱动。

延后项：
- 不做完整拓扑发现。
- 不做 CoE/SDO 事务。
- 不实现多平台后端并发落地。

### 阶段 3：拓扑发现、SII/ESI、配置计算

目标：
- 具备基础网络扫描、设备识别和配置计算能力。
- 打通从原始设备信息到可执行配置的静态链路。

核心任务：
- 实现从站发现流程，读取基础识别信息和拓扑顺序。
- 支持 SII 读取与基础解析。
- 支持 ESI 解析的最小可用子集，聚焦配置所需字段。
- 实现 Vendor ID、Product Code、一致性校验逻辑。
- 实现 FMMU/SM 自动配置计算，输出供运行时使用的配置对象。
- 为 `scan` 与 `validate` 命令准备可复用的库层接口。

验收标准：
- 可生成一份包含从站识别信息和配置结果的结构化扫描输出。
- 可对给定 ESI/SII 输入执行一致性校验并给出明确错误结果。
- FMMU/SM 计算结果可被后续运行时直接消费。
- 相关逻辑可在无真实网卡的夹具环境中回归测试。

延后项：
- 不实现完整对象字典高级校验。
- 不实现 DC 锁定。
- 不实现完整 CLI 交互体验，只保证库层能力完备。

### 阶段 4：ESM、PDO 周期通信、运行时编排

目标：
- 建立主站可运行的核心状态机和周期数据通路。
- 实现 Free Run 与 DC 方向兼容的运行编排框架。

核心任务：
- 实现 ESM 状态流转和错误回退路径。
- 实现位置寻址、别名寻址和逻辑寻址相关核心流程。
- 实现 PDO 周期通信的发送、接收和映射装载。
- 实现运行时调度器，统一周期循环、超时治理和背压控制。
- 加入 DC PLL 相关接口和状态推进框架，先形成集成点，再逐步补足算法细节。
- 建立抖动、重试、状态迁移的统计与诊断事件输出。

验收标准：
- 可在测试环境中完成从初始化到周期 PDO 交换的主流程演示。
- ESM 正常路径与关键错误回退路径具备测试覆盖。
- Runtime 能统一驱动 PDO 周期循环和事务推进，不依赖特定线程模型。
- Free Run 先达到可验证可回归，DC 路径至少具备接口和最小流程验证。

延后项：
- 不扩展到 `EoE/FoE`。
- 不追求全部平台同时达成同等级实时表现。
- 不做完整在线诊断产品化 UI。

### 阶段 5：CoE/SDO、CLI、诊断与文档完善

目标：
- 补齐非周期事务、产品化入口和面向使用者的文档体系。
- 让项目具备可操作、可验证、可演示的首版交付面。

核心任务：
- 实现 CoE/SDO Upload、Download 和 SDO Info 的事务推进机制。
- 完成请求队列、超时、重试、关联 ID 的 mailbox 事务模型。
- 实现 `moonecat scan`、`moonecat validate`、`moonecat run` 三类命令。
- 输出结构化诊断结果、日志、抓包或调试信息接口。
- 补全文档：架构说明、接口约束、测试方式、常见问题、典型示例。
- 评估并整理 Extism 宿主接入点，作为后续在线诊断方向的稳定边界。

验收标准：
- SDO 基础事务可独立测试，并能与运行时整合推进。
- CLI 三类命令具备最小可用体验，并直接复用库层能力。
- 用户文档和开发文档覆盖安装、运行、调试和扩展入口。
- 首版交付可以对外说明“支持范围、限制条件、下一步演进方向”。

延后项：
- 不承诺首版支持完整在线网页诊断工具。
- 不承诺首版提供所有平台的生产级网络驱动。
- 不引入申报书之外的大功能扩展。

## 4. 工作流拆分

### HAL

- 定义跨平台最小接口和错误模型。
- 优先完成 `Mock Loopback`，再落单一真实平台原型。
- 把网卡收发、计时、调度、文件能力保持为薄适配层。
- 为 Windows、RTOS/Embedded、Extism 预留后端接入点，但不提前做平台特化逻辑污染核心层。

### Protocol Core

- 实现 EtherCAT 帧、PDU、寻址、拓扑、错误模型。
- 实现 ESM 状态机及错误回退。
- 保持核心逻辑纯 MoonBit，可由测试直接驱动。
- 强制约束热路径少分配、可预分配和明确所有权边界。

### Mailbox & Config

- 实现 SII/ESI 解析和配置对象生成。
- 实现 FMMU/SM 自动配置。
- 实现 CoE/SDO 请求队列、超时、重试和结果关联。
- 保持非周期事务与 PDO 热路径解耦。

### Runtime

- 统一 PDO 周期循环、mailbox 推进、超时治理和背压控制。
- 支持 Free Run 优先落地，DC 同步逐步增强。
- 输出抖动、重试、状态迁移等可观测指标。
- 设计为与具体线程模型和宿主环境解耦。

### CLI/Library

- 对外只暴露一套共享核心实现。
- 库接口承担上层系统集成和嵌入式接入。
- CLI 固定聚焦 `scan/validate/run` 三类命令。
- 所有命令输出应基于库层结果对象，不在 CLI 层复制业务逻辑。

### 文档与测试

- 维护开发文档、用户文档和回归测试夹具。
- 以单元测试、夹具测试、协议流程回放和最小集成测试为主。
- 为后续 AI 协作补充模块职责、输入输出和验收标准。
- 把参考项目分析文档转成实现阶段的对照依据，而不是孤立材料。

## 5. 关键依赖与前后顺序

- `阶段 1 -> 阶段 2`：没有稳定包结构和 HAL/Core 接口，后续真实功能都会返工。
- `阶段 2 -> 阶段 3`：没有帧/PDU 闭环，无法可靠实现扫描、SII 读取和配置计算。
- `阶段 3 -> 阶段 4`：没有拓扑和配置对象，PDO 映射、状态迁移和运行时编排没有输入基础。
- `阶段 4 -> 阶段 5`：没有运行时主循环，CoE/SDO 事务和 `run` 命令只能停留在静态接口层。
- 可并行事项：
  - 文档与测试骨架可从阶段 1 开始持续推进。
  - `Mock Loopback` 和单元测试可与帧/PDU 编解码并行。
  - CLI 参数模型可在阶段 3 先定义，但命令真正完成依赖阶段 4 和阶段 5。
  - Extism 接入点可在阶段 5 前做接口预留，但不应阻塞核心主线。
- 明确约束：
  - CLI 不能早于核心库闭环被视为完成。
  - 多平台支持不能先于 HAL 最小接口稳定。
  - DC 不能阻塞 Free Run 最小可用版本交付。

## 6. 风险与缓解

### GC 带来的实时性抖动

- 风险：周期通信热路径若存在分配和临时对象堆积，会放大抖动并破坏实时性目标。
- 缓解：
  - 热路径采用预分配缓冲区和对象复用。
  - 阶段 2 起就约束帧/PDU 数据结构的分配行为。
  - 阶段 4 加入周期时延、抖动和重试统计。
- 验证动作：
  - 为核心循环建立压力回归。
  - 对比不同负载下的周期稳定性和错误率。

### 跨平台网卡接口碎片化

- 风险：平台差异过早渗入协议核心，会导致接口失稳和多端返工。
- 缓解：
  - 只暴露最小 HAL 能力面。
  - 优先做 `Mock Loopback` 和单一真实平台验证，再扩展其他后端。
  - 平台后端以适配器形式增量接入。
- 验证动作：
  - 以同一组核心测试驱动不同 HAL 实现。
  - 检查新增平台是否无需修改 Protocol Core 行为。

### 状态机逻辑死锁

- 风险：ESM、mailbox 和运行时推进之间若职责不清，容易出现卡死、超时恢复失败和错误回退缺失。
- 缓解：
  - 用显式 ADT/状态类型建模流转。
  - 保持事务推进和 PDO 热路径分层。
  - 为关键异常工况保留回放测试。
- 验证动作：
  - 覆盖 Init、PreOp、SafeOp、Op 以及错误回退路径。
  - 覆盖超时、重试、链路异常和配置不一致场景。

## 7. 近期优先级（前 30 天）

### 第 1 周

- 完成包结构、模块命名和分层接口草案。
- 定义 HAL 最小接口、错误模型、配置对象基础类型。
- 整理参考项目分析文档和申报书术语，形成统一命名表。

### 第 2 周

- 实现 EtherCAT 帧与 PDU 的最小编码解码能力。
- 完成 `Mock Loopback` 后端。
- 建立最小收发闭环测试。

### 第 3 周

- 实现基础扫描流程和从站识别结果对象。
- 补齐 SII 读取与最小解析链路。
- 输出首版结构化扫描结果。

### 第 4 周

- 实现 ESI 最小解析和 ESI/SII 一致性校验。
- 实现 FMMU/SM 自动配置的首版计算。
- 草拟 `scan` 与 `validate` 对应的库层接口和 CLI 参数模型。

### 30 天结束时的可交付物

- 一套稳定的分层骨架和命名体系。
- 一个可测试的帧/PDU 与 HAL 最小闭环。
- 一个可输出结构化结果的扫描与配置原型。
- 一份可供后续 AI 继续拆解 issue 的统一实施基线。
